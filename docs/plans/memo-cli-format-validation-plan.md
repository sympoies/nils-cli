# Plan: memo-cli format detection and validation metadata

## Overview
This plan adds non-blocking format detection and validation metadata to `memo-cli` using the existing
capture-first architecture, and resolves currently documented-but-unimplemented CLI options
(`add --at`, `report --tz/--from/--to`). `add` remains minimal and fast: users provide memo text
only by default, while `created_at` stays mandatory at storage level and is generated by the system
unless an explicit valid `--at` override is provided. Raw capture remains append-only in
`inbox_items.raw_text`; format and validation outcomes are recorded in the derivation layer
(Approach A) so they are versioned, replaceable, and safe to reprocess.

## Scope
- In scope: define and implement a common-format detection + validation pipeline (`url`, `json`,
  `yaml`, `xml`, `markdown`, `text`, `unknown`), persist results as derivation metadata, expose
  stable machine-readable fields in JSON mode, resolve spec-vs-implementation gaps for
  `add --at` and `report --tz/--from/--to`, and document/validate rollout behavior.
- Out of scope: blocking `add` on validation failures, network reachability checks for URLs,
  semantic/domain validation for payload content, cloud sync, and background daemon workers.

## Assumptions (if any)
1. `add` should continue to require only memo text input by default; timestamp remains mandatory in
   storage and is system-generated unless optional `--at` is provided with valid RFC3339 input.
2. Approach A is mandatory: raw data is immutable; classification/validation belongs to derivations.
3. Initial implementation prioritizes textual payloads from current CLI flow; binary classification
   is reserved as `unknown` until a byte-ingest path exists.
4. New metadata in JSON responses is additive and must not break existing consumers.

## Success Criteria
1. A pending item can be enriched with format + validation metadata without changing raw capture.
2. Common valid payloads (`url/json/yaml/xml/markdown`) are consistently tagged and validated.
3. Invalid structured payloads receive deterministic `validation_status` and `validation_errors`.
4. `add --at` is implemented with strict RFC3339 validation and safe default behavior.
5. `report --tz/--from/--to` supports explicit time-window reporting with deterministic precedence.
6. JSON and text outputs remain contract-stable with additive metadata only.
7. Required repository checks in `DEVELOPMENT.md` pass.

## Dependency & Parallelization Map
- Serial foundation:
  - Task 1.1 -> Task 1.2 -> Task 1.3 -> Task 1.4.
- Parallelizable after Task 1.3:
  - Task 2.1 (detector/validator engine) can run in parallel with Task 2.2 (contract+CLI shaping)
    after shared enums are frozen.
- Parallelizable after Task 1.4:
  - Task 2.5 (`add --at`) and Task 2.6 (`report --tz/--from/--to`) can run in parallel with
    metadata lane tasks if contract values are frozen.
- Serial integration lane:
  - Task 2.1 + Task 2.2 -> Task 2.3 -> Task 2.4.
  - Task 2.5 -> Task 2.6.
- Validation/documentation lane can parallelize late:
  - Task 3.1 depends on Task 2.4 + Task 2.6.
  - Task 3.2 can run with Task 3.1 once command behavior is stable.

## Testing Strategy
- Unit:
  - Detector/validator correctness, precedence rules, and failure normalization.
- Integration:
  - `add -> fetch -> apply -> search/report` roundtrip with metadata-enriched derivations and temporal options.
- E2E/manual:
  - Seed mixed real-world memo samples and verify machine routing via JSON fields.

## Risks & gotchas
- YAML/JSON ambiguity can cause unstable classification unless precedence is strict.
- XML and Markdown parser tolerance may over-accept malformed input without explicit guard rules.
- Additive JSON fields may still break brittle consumers that assume exact object shapes.
- Metadata stored only in payload/tags may need follow-up first-class columns if analytics scale.
- Timezone/DST boundary handling can produce off-by-one-day report windows without explicit tests.
- Allowing `--at` can introduce historical backfill ordering shifts that affect user expectations.

## Rollback plan
1. Stop metadata-producing apply jobs first; keep capture-only and read paths active.
2. Revert metadata-specific apply parsing/output changes while preserving raw and derivation history.
3. Keep existing `add/list/search/report/fetch/apply` baseline behavior and envelopes.
4. Re-run:
   - `cargo test -p nils-memo-cli memo_flow`
   - `cargo test -p nils-memo-cli agent_roundtrip`
   - `./.codex/skills/nils-cli-checks/scripts/nils-cli-checks.sh`
5. Re-enable metadata flow only after contract and runbook checks are green.
